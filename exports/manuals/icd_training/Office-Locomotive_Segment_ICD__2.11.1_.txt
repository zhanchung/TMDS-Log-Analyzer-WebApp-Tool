OFFICE TO LOCOMOTIVE INTERFACE
1.1 Scope
This document contains messages passed between the Positive Train Control (PTC) Office Segment and the Locomotive Segment, and 
messages passed between the Wayside Status Relay Service (WSRS) and the Locomotive Segment.
1.2 Message Format
Each message exchanged between the Office Segment/WSRS and Locomotive Segment consists of an application-layer header / wrapper which 
encapsulates the message payload data and includes a data check value.  The Edge Message Protocol (EMP) is utilized as the application-layer 
wrapper.  Segmentation of PTC messages exchanged between Office Segment/WSRS and Locomotive Segment is described in Table 1.
Table 1 PTC Office Segment/WSRS-Locomotive Segment Message
Field Size (Bytes) Description
Application Message Header Variable EMP Header
Message Body Variable Message payload.  See message definitions
Data Integrity Value 4 EMP Data Integrity Value

1.3.1 Message Number Field
The Message Number Field serves as a correlation ID which assists the Office Segment/WSRS and Locomotive Segment in associating a 
response with an initial message.  Specific requirements for populating the field are included in the on-board and office segment requirements.
1.3.2 EMP Source and Destination Fields
The source and destination address format and examples are defined in Appendix A of the EMP Specification.
1.3.3 EMP Data Integrity Field
HMACs for PTC messages between the locomotive and the BOS are generated using SHA-1-160 algorithm and the Locomotive OPK.  The result 
of the SHA-1-160 HMAC algorithm is a 160-bit value that is then truncated to 32-bits for insertion in the PTC messages.  This calculation and 
truncation is covered in the FIPS-198 and NIST 800-107 publications.
The keyed HMAC covers the EMP header fields and application message payload defined in this ICD.
1.3.4 Security Implementation
Details of the security implementation are in the I-ETMS Security Implementation Guide, Version 1.2, 2/17/2012.

1.4 Message Data
When fields are comprised of multi-byte quantities the byte order is presumed to be most significant byte first. For example, if a data quantity is 
represented in two bytes as a binary value of 259; the byte order for this field (as byte n and n+1) as it would appear in the message byte-stream 
is:
…..
2-byte quantity
byte n 1   (msb)
byte n+1 3   (lsb)
…..
The same convention applies to multi-byte quantities of other lengths.
Unless otherwise noted, all numeric fields in this document are unsigned.

Employee Middle Initial Variable Alphabetic Employee middle initial

[Unique error handling requirements]

Represents Train ID as reported by locomotive to office, 
ASCII text comprising 4 (alpha) bytes for RR (SCAC), left 
justified space filled and variable (alphanumeric) bytes, 
for train identifier.
Various Variable Alphanumeric Any field of variable size and type ‘Alphanumeric’ is 
defined to be ASCII numbers (0x30 through 0x39) and 
letters (0x41 through 0x5A and 0x61 through 0x7A) and 
spaces (0x20).
Various Variable Printable ASCII Any field of variable size and type ‘Printable ASCII’ is 
defined to be the printable ASCII characters (0x20 
through 0x7E) which are all of the characters defined for 
an ‘Alphanumeric’ field plus the following:
! “ # $ % & ‘ ( ) * + , - . / : ; < = > ? @ [ \ ] ^ _ ` { | } ~
Vendor Code 4 Alphanumeric GE = General Electric
NYAB = New York Air Brake
WRE = Wabtec Railway Electronics
