ABOUT THE MANUAL 
This manual is intended to explain how the ALSTOM CenTraCode® II-s (CTC2S) and 
CenTraCode® II-v (CTC2V) communication systems emulate the Genisys code system. 
This manual gives a detailed explanation of the software implementation of Genisys 
centralized traffic control coding systems. These systems operate on ALSTOM 
CenTraCode II hardware platform. 
The information in this manual is arranged into sections. The title and a brief description 
of each section follow: 
Section 1 – GENERAL DESCRIPTION:  This section gives general information on the 
CenTraCode II-s (CTC2S) and CenTraCode II-v (CTC2V) communication systems 
emulation of the Genisys code system. 
Section 2 – THEORY OF OPERATION:  This section gives general information on the 
operation of the CenTraCode II-s (CTC2S) and CenTraCode II-v (CTC2V) 
communication systems emulation of the Genisys code system. 
Section 3 – MAINTENANCE ACCESS CONSOLE:  This section describes the 
Maintenance ACcess (MAC) port that provides access to the operating system of the 
CenTraCode field module. 
Section 4 – DIAGNOSTIC MENUS:  This section discusses the Genisys specific 
menus available for the diagnostics and maintenance of the CenTraCode unit.  
Section 5 – CENTRACODE II-S HARDWARE:  This section describes the 
CenTraCode II-s hardware, including installation. 
Section 6 – CENTRACODE II-V HARDWARE:  This section describes the 
CenTraCode II-v hardware, including installation. 
Section 7 – TROUBLESHOOTING:  This section describes how to use various board 
LEDs to assist in troubleshooting the CenTraCode unit. 
Appendix A – CENTRACODE II-s SAMPLE CAA CONFIGURATION FILES:  This 
section provides sample CenTraCode II-s CAA configuration files for reference.  
Appendix B – CENTRACODE II-v SAMPLE CAA CONFIGURATION FILES:  This 
section provides sample CenTraCode II-v CAA configuration files for reference. 
Appendix C – COMMUNICATION SOFTWARE LIST:  This section lists the software 
programs that can be used for communication with the MAC. 
Appendix D – CENTRACODE II JUMPER CONFIGURATION:  This section provides 
the CenTraCode II-s and CenTraCode II-v jumper configurations.

MANUAL SPECIAL NOTATIONS 
In the ALSTOM manuals, there are three methods used to convey special informational 
notations to the reader. These notations are warnings, cautions, and notes. Both 
warnings and cautions are readily noticeable by boldface type two lines beneath the 
caption. 
Warning 
A warning is the most important notation to heed. A warning is used to tell the reader 
that special attention needs to be paid to the message because if the instructions or 
advice is not followed when working on the equipment then the result could be either 
serious harm or death. The sudden, unexpected operation of a switch machine, for 
example, or the technician contacting the third rail could lead to personal injury or death. 
An example of a typical warning notice follows: 
WARNING 
DISCONNECT THE MOTOR ENERGY WHENEVER THE GEAR COVER 
IS REMOVED. OTHERWISE, THE SWITCH MACHINE MAY OPERATE 
UNEXPECTEDLY AND POSSIBLY CAUSE PERSONAL INJURY 
Caution 
A caution statement is used when an operating or maintenance procedure, practice, 
condition, or statement, which if not strictly adhered to, could result in damage to or 
destruction of equipment. A caution statement is also used when personnel could be 
surprised if shocked by a circuit operating at a low current. A typical caution found in a 
manual is as follows: 
CAUTION 
Turn power off before attempting to remove or insert circuit boards into a 
module. Boards can be damaged if power is not turned off. 
Note 
A note is normally used to provide minor additional information to the reader to explain 
the reason for a given step in a test procedure or to just provide a background detail. An 
example of the use of a note follows: 
NOTE 
A capacitor may be mounted on the circuit board with a RTV adhesive. 
Use the same color RTV.

1. SECTION 1 – GENERAL DESCRIPTION 
1.1. SCOPE OF MANUAL 
The purpose of this manual is to explain how the ALSTOM CenTraCode® II-s (CTC2S) 
and CenTraCode® II-v (CTC2V) communication systems emulate the Genisys code 
system. This manual gives a detailed explanation of the software implementation of 
Genisys centralized traffic control coding systems. These systems operate on ALSTOM 
CenTraCode II hardware platform. 
1.2. OVERVIEW 
The Genisys code system is a system of communications that allows for the movement 
of information. A Computer-Aided Application (CAA) program provides all the location-
specific items, such as the location's code charts. It is used to program the system to 
interface with the existing application cabinets. It can also be programmed to replace 
the non-vital application logic functions performed in these application cabinets. The 
Maintenance ACcess (MAC) port assures all access needed to the code unit. A menu-
driven command interface is provided to clearly identify available commands and 
options. 
To access MAC, any VT-100 compatible terminal may be used or, with the proper 
software, an IBM PC or compatible can also be used. Generally, a battery-operated 
laptop is the most convenient terminal device for field work. It is recommended for use 
with the code unit because logging modes that show the codeline's traffic can be 
requested. Using a personal computer as a display terminal allows the capture of 
logging displays to disk files for further analysis.  
Appendix A and Appendix B provide sample CAA configuration files for both the 
CenTraCode II-s and II-v systems. Appendix C lists the available software packages 
that can be used for communication with the MAC. Appendix D gives the jumper 
configuration for both systems.

1.3. CAPABILITIES/LIMITATIONS 
The CenTraCode II (CTC2) emulator replaces code steppers directly. It provides the 
data path to move controls and indications from one location to another at a preset rate 
of transmission.  
1.3.1. CenTraCode II-s (CTC2S) System 
This stand alone unit interfaces with an application logic cabinet or the function relays. 
The basic CPU module contains solid state, parallel I/O consisting of 24 inputs and 16 
outputs and 4 serial I/O ports providing RS232 voltage level support. 
Indications must be connected to the unit directly. All latching and station mapping is 
internal. 
The unit provides non-vital logic applications without an external logic cabinet. 
1.3.2. CenTraCode II-v (CTC2V) System 
This unit provides the same non-vital logic functions as the CenTraCode II-s system but 
within the VPI® Vital Processor Interlocking system chassis. The CTC2V unit, also 
referred to as CSEX, receives indication data from and passes control data to the VPI 
application logic through a dual port memory interface (DPRAM). The basic module 
supports six serial ports; ports 1 and 2 can be strapped to support RS422 or RS423. 
Ports 3, 4, and 5 support RS422 and port 6 supports the current loop. 
For both hardware platforms the timing and configuration parameters are adjusted by 
using console commands. No mechanical adjustments are needed. 
1.3.3. Configuration Limits 
The following are the maximum configuration parameters of the standard Genisys 
protocol: 
• 1,792 controls or indications 
• 255 unique station address (address x00 is reserved)

2. SECTION 2 – THEORY OF OPERATION 
2.1. GENERAL 
This section contains the theory of operation of the CenTraCode II-s (CTC2S) and 
CenTraCode II-v (CTC2V) communication systems emulation of the Genisys code 
system. 
2.2. PROTOCOL DESCRIPTION 
The Genisys system protocol functions on any port of the CTC2S or CTC2V systems.  
In the explanations that follow the letter 'x' followed by a number or letters A-F indicates 
hexadecimal values. 
2.2.1. Communication Addressing 
All field units allow the addressing of messages to range from x01 to xFF (address x00 
is reserved for common controls (x indicates hexadecimal notation). 
All units allow message content for controls or indications in the range of x00 to xDF. 
Controls are sent from Master to Slave, Indications are sent from Slave to Master. 
Controls or Indications xE0 and beyond are reserved for status type information.

2.2.2. Message Structure 
All messages have a common structure. Messages start with a command byte, xF1 - 
xF3, or xF7 - 0xFE and are terminated with a terminator byte 0xF6. The command byte 
is followed by the address of the destination station. Some messages contain a body 
before the terminator byte; see the next section for details. To avoid reserved byte 
aliasing, any byte other than the command or terminator in the xF0 to xFF range is split 
and sent to as two bytes. 
Example: 
 Byte  Sent As 
 xF3  xF0 x03 
2.2.3. Transit Message Types 
The three transit message types: Acknowledge, Change, and Control Checkback are 
described below. 
Acknowledge – Informs the remote unit that the message was received and accepted.  
  FORM : xF1 st xF6 
Change – Sends changed indications to the remote unit. 
  FORM : xF2 st {bn bi} cl ch xF6

Control Checkback – When control checkback is enabled in the mode byte (explained 
below) and a control is received from the master, the message is echoed back to the 
master. No action is taken until the master confirms the message by sending an 
execute command.  
  FORM:  xF3 st {bn bi} cl ch xF6 
• xXX  are hard coded hexadecimal  values that never change 
• {} indicates that byte pair may be repeated as needed 
• cl indicates crc16 lower byte 
• ch is crc16 upper byte  
• bn is byte number, either control or indication byte number 
• bi is byte information 
• st indicates station address 
• bi[x] is byte x of indication buffer 
2.2.4. Receive Message Types 
The seven receive message types: Reserved, Common Controls, Acknowledge, Poll, 
Control, Recall, and Execute are described below. 
Reserved – Message types xF7 and xF8 are reserved for future use. 
Common Controls – Can be sent by master to any one station or broadcast to all 
stations (address = x00). This message has access to change only bytes 0 and 1 of the 
controls.  
  FORM:  xF9 st {bn bi} cl ch xF6 
Acknowledge – Informs the master unit that the message was received and accepted.  
  FORM:  xFA st cl ch xF6 
Poll – Requests that the slave report any changes or, if no changes have occurred, 
then generates an acknowledge. 
This command takes two forms depending on the mode byte.  
  FORM:  xFB st cl ch xF6 - in secure mode 
                     xFB st xF6       - in non-secure mode

Control – Sends controls to the slave. 
  FORM:  xFC st {bn bi} cl ch xF6 
Recall – Requests slave to send a change message containing all indications. 
  FORM:  xFD st cl ch xF6 
Execute – In control checkback mode, if the slave response is valid, this command 
causes the slave to use the controls. 
  FORM:  xFE st cl ch xF6  
2.3. MODE OF OPERATION 
The Genisys protocol calls for a Master unit to poll all known stations in a system 
periodically. When a slave receives a poll with its address, the slave responds. If the 
slave has outstanding indications, the slave responds with an indication message. The 
Master then either sends a control message to the slave or acknowledges the slaves 
indication message. The slave can continue to send indication messages as long as the 
Master continues to send messages to it. When the slave has no further messages for 
the Master, or in response to a poll, the slave can return an acknowledge to the Master. 
The Genisys emulation operates in the slave mode only. Upon powerup, the system 
sets the mode byte to secure poll, no checkback, no common controls, and data base 
not complete. When a poll with the matching station address is received, the slave 
responds with a message containing all indications, and appends the mode byte (xE0) 
onto the end. If the Master returns with another message, the slave then sends any new 
indications, or acknowledges the Master to show no indications are left.

2.4. DATA HANDLING 
Data flow is controlled in two main tasks. 
1. I/O Handler 
2. Receiver 
Figures 2–1 and 2–2 provide control logic and indication data flow charts. 
2.4.1. I/O Handler 
The I/O handler task checks the application logic buffer to determine when a new buffer 
is ready to be sent. When a new buffer is detected, the information is copied into a local 
buffer and the application buffer is released. The buffer is then flagged for transmission. 
The local buffer that was copied and holds the application information is maintained and 
held ready to be sent if a recall should be required. 
2.4.2. Receiver 
The receiver tasks checks the incoming data stream to see if there is a message for this 
unit. When a message address of the proper value is detected the message is checked 
for correct checksum (if applicable) and termination information. When all criteria are 
met, the message is passed to the application logic. Flags are then set to indicate what 
type of message was received. The appropriate response to the message is triggered to 
the transmitter.

2.5. CONFIGURATION BY CAA PACKAGE 
Many of the operating features of the system can be configured from the CAA. Each 
option is discussed individually. For more detailed information about entering records in 
the CAA program, please refer to Publications 2326C and 2086C. For a step by step 
configuration and an example of the CAA input files, see the "Configuration -- Sample 
CAA Input File" in Appendix A and B.  
2.5.1. Serial Port 
This option identifies the port on which the Genisys Emulation is run. This option must 
be specified in the SERIAL PORT Record.  
2.5.2. Configuration File 
The configuration data file is specified using this option. If none is specified, then default 
information is loaded. The CAA CONFIGURATION FILE Record defines the file name. 
The configuration file is created and modified by the configuration file editor "GENCFG" 
which is further explained later in this section. 
2.5.3. Baud Rate 
This option specifies the baud rate for the code system. This option is set in the 
DEFAULT BAUD RATE Record. This option does not have to be explicitly specified. 
2.5.4. Data Format 
This format represents three different options:  data bits, stop bits, and parity. 
2.5.5. Control Message 
The CONTROL Record contains the location selection address (Location ID) and the 
number of control bits. The station selection address may also be included with the 
location selection to perform internal logic. The control bits may then be assigned.  
2.5.6. Indication Message 
The INDICATION Record contains the station address (Location ID) and the number of 
indication bits. The indication bits may then be assigned.

2.5.7. Special Message 
The SPECIAL CONTROL Records are optional. They contain the information required 
for improved interface to the Genisys protocol.

2.6. LOCAL CONFIGURATION EDITOR 
The configuration file is generated by the "GENCFG" editor. This file provides 
installation specific timing data. The editor prompts you for values of the timing 
parameters required to properly operate the code unit. This information is stored in a file 
that is accessed by the CAA program when generating the application EPROMs. The 
data is then stored in tables for the unit usage. These parameters can also be modified 
using the MAC interface.  
The editor stores the data in ASCII representation of the hexadecimal values. The file 
created by the editor must be entered in the CAA input files for the CAA program to 
access it. The proper format to enter the file name is using the following CAA record: 
 CONFIGURATION FILE = filename 
If this record is not defined, then the CAA program generates default data for the 
installation. 
2.6.1. Using GENCFG 
This editor is used to tailor the timing parameters used by the protocol to the particular 
application. The default values are assigned by ALSTOM in accordance to published 
specifications. The following describes how to use the editor. 
1. The editor is invoked by typing GENCFG ( optional filename .LPC) at the prompt line. 
The initial welcome screen is displayed. Press a key to acknowledge the copyright 
notice. 
2. Enter the name of the confi guration file to use. The configuration file should always 
use the file extension .LPC. If you are creating a new file, enter the new name and 
press "Y" to create it. 
3. The editor loads the values from the file . If the file size is incorrect, a warning 
message is displayed. Continuing the process loads whatever values appeared in 
the file. After loading is complete, the entire file content is displayed on the screen. 
4. To change a particular parameter; one of these methods can be used: 
− Press the space bar to highlight t he parameter. Once the parameter is 
highlighted, press [ENTER]. 
− Press the "Alpha" letter identifyi ng the parameter then press [ENTER].

5. When the parameter is sele cted, it displays the current file value in parenthesis and 
prompts for the new value. The following are the acceptable entries: 
− [ESC] to load the defined default value. 
− The new decimal value of the parameter  followed by [ENTER]. Note that if no

2.6.2. Genisys Emulation C onfiguration Parameters 
Rts to Cts – Defines the maximum amount of time after asserting Rts to wait for Cts. If 
Cts does not become true after this time delay, Cts is ignored and transmission takes 
place. If this is zero, the port waits 3 seconds and aborts the message transmission. 
Hold Rts – Defines the amount of time to hold Rts true after the last byte of a message 
has been transmitted. 
Multi Drop – If true, causes Rts to assert at the beginning of transmission and deassert 
at completion. If false, Rts is asserted all the time. 
Cts – If true waits for Cts to become true before transmission begins. If false, ignores 
Cts. 
Invert Cts/Invert Rts – If false, the asserted level of that control is HIGH. If true, the 
asserted level is LOW.
