Genisys Protocol


The TMDS wayside server supports non-secured polling for the Genisys protocol. The implementation and description of the protocol is listed below. The data is always framed with a header and footer byte. The header byte describes the mode or type of packet being transmitted or received. The footer is the termination of the message, which is always a 0xF6. The second byte in the message is always the station address (0x1 to 0xff). 
There will never be a byte value greater than 0xF0 between the header and footer byte. If a byte has a higher value of 0xF0 between the header and footer it will be represented by two bytes. For example, if there is a data byte that needs to be transmitted in the frame of 0xFF, it will be represented by 0xF0 & 0x0F (e.g. <F0><0F>). The 0xF0 byte in a data frame is a trigger that there are two bytes used to represent one value. 
There is also a special byte call the 0xE0 byte, which is found in the office and field messages typically during initialization only and indicates how the polls and controls interact between the office and field. We only support non-secure polling (this means there is no CRC in the poll messages from field) and we support control check back (this means the field echoes the control back to the office, the office checks the echo and then tells field to execute 0xFE). 

All valid genisys messages start with one of the following two-character Office or Field Header bytes:
Office Header Bytes 
0xFA – Acknowledge 
0xFB – Poll 
0xFC – Control 
0xFD – Recall 
0xFE – Execute 
Field Header Bytes 
0xF1 – Poll Response No Data 
0xF2 – Poll Response with Data
0xF3 – Control Response with Data (only used if “control check back” is enabled)
Data frames dissected with legend 
• Bytes in yellow are the CRC-16 
Bytes in blue are the control point address (in hex)
• Bytes in gray are the data byte positions (in hex - starting at zero… 0, 1, 2, 3, etc)
• Bytes in green are the values of the data bytes for their relative positions (1-8)
• Bytes in red are the initialization bytes.  <E0> <07>  the field uses non-secure polling and control check back, if these values are <E0> <05> the field  uses non-secure polling and NO control check back

All valid genisys messages end with <F6>
Additional characters may be present after the F6, but they are ignored.

Field Response			<F1><12><F6>

A common setting is for field generated indications caused by a change of state to only send the data that has changed.  In this case, the value of word 2 changed to <00>.  Office Recalls will still respond with a full indication string.

Office Poll			<FB><08><C3><3D><F6>
Field Response			<F2><0B>02><00><E0><07><78><0C><F6>	
Office Acknowledgment 	<FA><0B><03><67><F6>
Field Indication no/data	<F1><0B><F6>

Another common setting is for non-secure polling and NO control check back.  Initialization bytes <E0> <05> are used when in this mode.  <FE> Execute is not used.

Office Control (no check back)	<FC><01><00><01><01><00><02><00><E0><05><41><C6><F6>

Indication and control data bytes are in hex.  They need to be converted to binary in least significant digit order.
Each byte is eight digits long.  If the binary conversion is less than eight digits, leading zeros must be added.
For example…
01 Hex = 01 Binary -> eight digits long = 00000001 -> least significant digit order = 10000000
59 Hex = 01011001 Binary -> no leading digits needed -> least significant digit order = 10011010


Field Indication data <F2><0B><00><01><01><59><02><40><03><00><04><00><E0><07><78><0C><F6>
<F2> = Field Indication with data
<08> = Station address (in hex, convert to decimal)
<00> = Word 0 (bits 1-8)
<01> = Word 0 data = 01 hex = 01 binary + leading digits = 00000001 -> converted to least-significant-digit order = 10000000
<01> = Word 1 (bits 9-16)
<59> = Word 1 data = 59 = 01011001 -> 10011010 (bits 9, 12, 13, and 15 = TRUE)
<02> = Word 2 (bits 17-24)
<40> = Word 2 data = 40 = 01000000 -> 00000010 (bit 23 = TRUE.  Bit 23 may also be referred to as “Word 2 Bit 7”)
<03> = Word 3 (bits 25-32)
<00> = Word 3 data = 00 = 00000000
<04> = Word 4 (bits 33-40)
<00> = Word 4 data = 00 = 00000000
The indication string would be all indication values together:
1000000010011010000000100000000000000000

In the next example, the value of indication word 3 is F3.  
Values larger than F0 are represented as two bytes.
<F2><0B><00><01><02><59><03><F0><03>><04><00><E0><07><78><0C><F6>
<03> = Word 3 (bits 25-32)
<F0> = Word 3 data byte 1 = F0 = 00001111
<03> = Word 3 data byte 2 = 03 = 11000000
Word 3 data is then combined = 11001111

Control Data would be read the same way.

When converting hex to binary, I use an iOS application called NerdCalc.
Leading zeros are automatically added.  
To show least significant digit order, I simple disable screen rotation, and turn the phone upside down.

The windows calculator can be used.  Base conversion can be performed if it is set to “Programmer Mode”.
Make sure you reverse the conversion so it will be in least significant digit order.  Be careful with the calculator in older versions of windows as leading digits are not added.

NOTE:
Genisys numbering STARTS with WORD-0.
Word-0 contains bits 1-8.
Word-0 Bit-1 is going to be displayed in the CodeServer as Bit-1
Word-1 contains bits 9-16.
Word-1 Bit-1 is going to be displayed in the CodeServer as Bit-9.
Prints may contain a different numbering scheme.  For instance, BRC/Alstrom prints start with Word-1 Bit-0… so Word-1 Bit-7 on the print is the same as genisys Word-0 Bit-8.

When setting up new locations, the number of control and indication bits needs to match the value programmed in the field unit.
If the number of controls does not match, the control will most likely fail.
If the number of indications does not match, unexpected behavior may occur.
