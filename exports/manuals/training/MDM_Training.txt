=== Slide 3 ===
Changes have been made to MDM to address memory and performance issues.
The overall design and many components were retained, but intertwined UI/Business Logic was eliminated.
Functionality was separated into a new architecture where the Back Office Control (BOC) provides the UI, and a much more lightweight service-style system provides the Business Logic.
Test results have been
very
positive for memory usage as well as performance.
Note: This
document makes reference to “Admin Client” in several places. This applies to either of the applications “TMDS
AdminClient
” or “TMDS WebApp”. Both work with MDM and, for our purposes in this document, have the same functionality needed.
Overview

=== Slide 4 ===
MDM (since v5.2) can now be installed as a service. The basic functionality to this is trivial. Simply open up a command prompt running as Administrator, navigate to the MDM install directory, and execute the following command:
> MobileDeviceManager.Service.exe install
Use the command “uninstall” in the same fashion to uninstall it from the service system.
It can be controlled via the Services administration tool or in the same fashion as above using the commands “start”, “stop”, or “restart”.
Of course, you can just double-click the executable or start it from the command prompt as you would with any application. It will run as a standalone console application or can be installed to run as a service.

=== Slide 11 ===
The server has a maximum number of concurrent downloads setting. If reached, additional download requests are queued until a download completes and then the first on the queue is serviced. A visual display of this has been added to the lower-right corner of the BOC showing how many concurrent downloads, a percentage-based progress bar indicating how close the server is to the maximum, and the number of downloads which are queued. As the progress bar fills up, it changes color. Less than 75% it will be green, at 75% it will turn yellow, and at 85% it turns red. Having downloads queued is not necessarily a bad thing, this is just a way to reduce your resource load and protect against overloading the system. It can be controlled via the MDM config item
MaximumConcurrentActiveDownloads
in the
TransferSettings
section. A download is considered active if the 2104 has not been received and there was 2103 traffic within the last 30 seconds. The timeout value for a download is configurable on the server, the default is 5 minutes.
Back Office Control (BOC): Download Queue Status

=== Slide 16 ===
MDM now starts up similarly to BOS insofar as it has an Initialization stage.
During Initialization, MDM checks and verifies the configuration settings to make sure they are within any required ranges, correct values, or specific value sets. Previously, MDM would try to guess if you provided invalid input, but this is a mistake when we’re operating with a critical server, so those types of errors are now logged and required to be corrected (e.g. If you set “
DbConnectionTimeout
” to “A” instead of a number, you’ll get an error during Initialization). The reason these are now errors is most configuration item “guesses” have consequences down the road, so we simply ask they be fixed up front.
So, if your MDM is not making it through Initialization, please check the logfile for Error messages and correct the issues.
It is suggested that one routinely looks through Initialization logs for Warnings as well, since those will indicate possible issues which could cause aberrant behavior later.
Anatomy of the Server: Initialization

=== Slide 19 ===
Uploads are limited by the configured number of locomotives which can have an Upload Session at once. This puts a finite cap on the amount of bandwidth used for Uploads. Downloads, on the other hand, are only limited by the maximum number of simultaneous downloads. This value is configurable and will vary per customer depending on their available resources and desired configuration. These two items allow the customer to define the relationship between Upload and Download priority.
The basic formula is a locomotive will send a 2100 providing data about its current repositories and requesting MDM to provide the data it
should
have. If the locomotive finds discrepancies, it will then start requesting files to download (via the 2101) until it has satisfied the data MDM provided. Another 2100 is then sent to indicate everything is completed.
Once MDM has received (and agrees with) a 2100 which indicates all items are synchronized (or it receives a 2105 indicating the locomotive has no room to download data), it will add that locomotive to the “Upload Wait Queue”. Once a position is available (based on the
MaximumLocomotivesUploading
config item), the next locomotive in the wait queue begins its “Upload Session”. For a given “Upload Session”, MDM will query the locomotive for files on each of its slices until the locomotive indicates it has no more files to upload. This concludes an “Upload Session” which allows a new locomotive to begin.
Setting the Locomotive Group of a locomotive to Inactive will cause all Downloads and Uploads to be cancelled. Active or Inactive locomotives are always serviced for 2100/1100 message responses.
Anatomy of the Server: Uploads and Downloads

The Foreign Loco Queue (when enabled) is a mechanism which allows for N download slots for each foreign SCAC. Once a queue fills up for a given SCAC, new download requests are refused until there is a slot open again. This

=== Slide 22 ===
MDM now properly sets the Quality of Service and Time to Live settings on EMP messages!
The MDM requirements dictate that for the 1100, 1101, 1102, 1103, 1104, and 1105 messages, the QoS and TTL settings will mirror the 21xx message to which it is responding. For example, if a 2101 was received indicating a Priority of 7, the 1101 response would be sent with the same Priority.
For unsolicited messages, MDM follows the settings from the configuration file “EmpMessageSettings.xml” from the installation directory. The default XML file is based off on the PTC BackOffice Loco ICD 4.0 table 15.1, but these settings are customer configurable.
The XSD is provided (EmpMessageSettings.xsd) if you wish to edit the XML file. The structure is analogous to the table provided in the PTC BackOffice Loco ICD 4.0. For each message, you can provide a “
datarow
” which defines the data settings according to the “
pollrate
” or “
qostype
” (as defined in columns “Poll Rate” and “Type” in table 15.1 of the ICD). For example, the 1101 message has two different data sets based on the QoS Type of “File Transfer” or “Urgent”. This translates to ‘
qostype
=“
file_transfer
”’ or ‘
qostype
=“urgent”’. Any of the “data” element options can be user-configured in accordance with valid settings as defined by the Edge Message Protocol specification.
It should be noted that for MDM, these settings only apply to
unsolicited
messages. Any message responding to a message from the locomotive will use the locomotive’s settings.
Anatomy of the Server: QoS and TTL Settings

=== Slide 23 ===
Since MDM 5.0.0, SQL Notifications have been required to support changes to Loco Group data from the database. If you did not have notifications enabled, then you got lots of errors in the logs and a significantly delayed response time when making Loco Group changes via Admin Client.
“
LocoGroupIdDbCacheUseSqlDependency
” is a new Boolean config item which allows you to disable using notifications and the subsequent errors. We absolutely recommend SQL notifications be set up so you will get instant updates to Loco Group data changes, but if you don’t feel the need for this you now have the option to turn it off.
“
LocoGroupIdDbCacheUpdateInterval
” lets you set how often MDM will automatically update the Loco Group data cached in its memory. If you are
not
using SQL notifications, we recommend this be at 15 seconds. If you are using SQL notifications, you can set it to a bigger value such as 60 or 120 as it becomes just a fallback to ensure the cache is being updated. This cache eliminates a significant number of database queries for every message exchange, so even if you are updating the cache every 15 seconds this is a huge efficiency over querying every single message.
If you have notifications enabled, MDM will test it to make sure it works during initialization. If it does not, then it will disable it and log warnings. If you have warnings about this failing, they should be addressed as updates to Loco Group data will likely be significantly delayed (depending on your update interval).
Anatomy of the Server: SQL Notifications Change
